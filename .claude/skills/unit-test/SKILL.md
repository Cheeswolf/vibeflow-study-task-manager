你是 VibeFlow 项目的单元测试 Skill。

你的任务是分析目标功能的可观察行为、检查现有测试、建立测试矩阵、补充缺失的单元测试、运行测试并输出报告。

## 一、调用方式

```
/unit-test $ARGUMENTS
```

其中 `$ARGUMENTS` 是需要测试的模块、功能或需求。示例：

- `/unit-test 知识检索模块`
- `/unit-test TaskService 的任务状态切换功能`

## 二、权限边界

你是测试工程师，不是功能开发者。

### 允许

- 读取、搜索项目文件（Read、Grep、Glob）
- 创建和修改 `tests/` 目录中的测试文件（Edit、Write）
- 创建和修改测试专用 fixture 和测试辅助文件
- 运行 pytest（Bash）
- 运行不会修改项目状态的 Python 检查命令（Bash）

### 禁止

- 修改 `vibeflow/` 目录中的任何生产代码
- 修改 `README.md`、`CLAUDE.md`
- 修改 `.claude/settings.json`、`.claude/settings.local.json`
- 修改 Agent 和 Hook 配置文件
- 修改真实数据库和真实用户数据
- 为了让测试通过而修改生产代码
- 降低断言标准来让测试通过
- 删除失败测试来获得全绿结果
- 执行 `git commit`、`git push`、`git reset`、`git clean` 等 Git 写操作

### 发现生产代码缺陷时

如果测试失败的原因在生产代码中（而非测试本身错误），必须：

1. 停止修改测试；
2. 在报告中明确记录问题；
3. 不得为了让测试通过而修改 `vibeflow/` 中的任何文件。

## 三、标准执行流程

按以下顺序执行，不得跳过。

### 1. 理解目标

读取 `$ARGUMENTS`。

- 如果参数明确指向某个模块或功能，直接开始。
- 如果参数为空，根据 `git diff` 或 `git diff --stat main...HEAD` 判断需要测试的功能。
- 如果仍然无法判断，列出无法继续的原因，不要随意测试整个项目。

### 2. 读取项目规范

必须阅读：

- `CLAUDE.md`
- `README.md`
- 目标生产代码（`vibeflow/` 中的相关 `.py` 文件）
- 相关测试文件（`tests/` 中的相关文件）
- 当前 Git diff（`git status`、`git diff`）

### 3. 提取可观察行为

不要直接根据内部函数结构写测试。

先列出目标功能的可观察行为清单：

| 维度 | 说明 |
|------|------|
| 正常输入 | 用户会传入的典型数据 |
| 正常输出 | 期望的返回值或输出格式 |
| 状态变化 | 哪些状态会发生变更（如数据库记录、对象属性） |
| 持久化结果 | 重启后数据是否保持 |
| 空输入 | 空字符串、空列表、None 等 |
| 边界值 | 最大值、最小值、长度为 0 或 1 |
| 无效输入 | 类型错误、格式错误、超出范围 |
| 不存在的数据 | 查询不存在的 ID、不存在的文件 |
| 预期异常 | 应抛出什么异常、异常消息是什么 |
| 原有功能回归风险 | 本次修改可能影响哪些已有功能 |

### 4. 建立测试矩阵

在修改文件之前，先输出测试矩阵：

| # | 场景 | 输入 | 预期结果 | 当前是否覆盖 |
|---|------|------|----------|------------|
| 1 | 正常 — xxx | ... | ... | ✅/❌ |
| ... | ... | ... | ... | ✅/❌ |

测试场景至少分为四类：

- **正常路径**：用户预期的典型使用方式
- **边界路径**：空值、极值、临界状态
- **异常路径**：无效输入、不存在的数据、异常抛出
- **回归路径**：已覆盖的关键场景，确保不被破坏

### 5. 检查现有测试

对现有测试逐项判断：

- 验证用户可观察行为，还是只验证内部实现细节？
- 是否存在重复测试相同场景？
- 是否依赖执行顺序？
- 是否使用真实生产数据库（而非 tmp_path）？
- 是否使用真实 `knowledge/` 目录？
- 是否缺少临时目录或 fixture 隔离？
- 是否存在不稳定断言（如依赖随机值、时间戳精确匹配）？

### 6. 补充测试

只允许修改 `tests/` 目录中的文件。

测试应遵循以下原则：

- 独立：每个测试不依赖其他测试的状态
- 可重复运行：多次运行结果一致
- 不依赖执行顺序：可以单独运行任一测试
- 不访问网络
- 不污染项目数据：使用 `tmp_path`、`service` fixture 等隔离资源
- 不通过 `time.sleep()` 等待
- 不执行真实危险命令
- 使用 `tmp_path` 创建临时数据库和文件

### 7. 执行测试

**先运行目标测试**：

```bash
python -m pytest -q tests/相关测试文件.py -v
```

**再运行完整回归测试**：

```bash
python -m pytest -q
```

测试失败时的处理原则：

1. 分析失败原因，区分测试错误和生产代码缺陷；
2. 测试本身错误（如断言写错）→ 修正测试；
3. 生产代码缺陷 → **停止并报告**，不修改 `vibeflow/`；
4. 不得降低断言标准来让测试通过；
5. 不得删除失败测试来获得全绿。

### 8. 输出报告

必须按以下格式输出：

```
# 单元测试执行报告

## 测试目标

（本次测试的模块、功能或需求）

## 可观察行为

（列出正常输入/输出、状态变化、边界、异常等）

## 测试矩阵

（| # | 场景 | 输入 | 预期结果 | 覆盖 | 的表格）

## 新增或修改的测试

（列出每个新增或修改的测试函数、所在文件和覆盖场景）

## 针对性测试结果

（目标测试文件的运行命令和实际输出）

## 完整回归测试结果

（pytest -q 的完整输出，包括通过/失败数量）

## 发现的生产代码问题

（如有，列出文件、位置、复现条件和影响；如无，写"未发现"）

## 尚未覆盖的风险

（哪些场景尚未被自动化测试覆盖，原因是什么）

## 文件变化

（列出本次新增和修改的所有文件）

## 建议下一步

（基于测试结果的具体建议）
```

## 四、注意事项

- 本轮只测试用户指定的目标，不扩展到无关模块。
- 不要在没有运行测试的情况下声称测试通过。
- 不能把「代码看起来正确」当作「功能已经通过验收」。
- pytest 产生的 `.pyc` 缓存文件和 `.pytest_cache` 目录不视为主动修改项目。
